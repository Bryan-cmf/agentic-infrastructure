## 🌏 完整說明（選擇你的語言）

### 🇹🇼 繁體中文

**痛點：** 每次做完才發現「早知道就不要這樣做」——80% 的時間花在勘探而非建設、重複踩同樣的坑、無意義的工作在事後才被識別。我們缺少一個在動手之前就預判所有可能路徑的機制。就像下棋不先算三步、打仗不做戰前推演——注定會輸在效率上。Meaning 技能解決了「事後回顧」的問題，但缺少「事前預判」。

**方案：** 多情境發散式路徑預判。接到複雜任務後，不直接執行——先展開所有可能的執行路徑（3-5 條），每條路徑進行：(1) 瓶頸預判：哪裡會卡住？(2) 踩坑預判：這條路徑的已知失敗模式？(3) 浪費預判：勘探/建設比率會是多少？(4) 風險評分：綜合預期浪費機率。最後輸出最優路徑推薦 + 所有路徑的風險地圖。

**核心機制：Pre-mortem（事前驗屍）——「假設這條路徑已經失敗了，為什麼？」**

**效果：** 每條複雜任務在執行前經過多情境預判。勘探/建設比率在事前就被評估。已知踩坑模式被預先標記。用戶可以選擇最優路徑或自定義混合路徑。

## 概念

> **「假設這條路徑已經失敗了——為什麼？」**

Pre-mortem（事前驗屍）是軍事和工程領域的核心方法論：在行動開始之前，想像它已經失敗，然後回溯找出所有可能導致失敗的原因。Agent Previsor 把這個方法做成了一個技能。

## 觸發方式

| 方式 | 說明 |
|------|------|
| 自動 | 複雜任務（多步驟 / 高不確定性 / 涉及多技能）→ 自動觸發 |
| 手動 | `/previsor`、`「預判」`、`「幫我分析路徑」` |

## 預判範圍（全部）

| 預判類型 | 說明 |
|---------|------|
| 🔀 **流程瓶頸** | 哪個步驟會卡住？依賴什麼？需要的資源？ |
| 🕳️ **已知踩坑模式** | 這條路徑的歷史失敗記錄？lessons 中的相關教訓？ |
| 🗑️ **無意義勘探** | 這條路徑的勘探/建設比率預估？是否 >60%？ |
| ⏱️ **時間浪費** | 哪個步驟有高機率變成重複勞動？ |

## 執行流程

### Phase 1：發散（展開所有可能路徑）

```
收到複雜任務
    │
    ├── 路徑 A：最直接的方法（default path）
    │   └── 瓶頸：？ 踩坑：？ 浪費率：？
    │
    ├── 路徑 B：避開已知瓶頸的替代方法
    │   └── 瓶頸：？ 踩坑：？ 浪費率：？
    │
    ├── 路徑 C：極簡方法（砍掉所有非必要步驟）
    │   └── 瓶頸：？ 踩坑：？ 浪費率：？
    │
    ├── 路徑 D：反向方法（從結果倒推）
    │   └── 瓶頸：？ 踩坑：？ 浪費率：？
    │
    └── 路徑 E：詢問用戶（最安全）
        └── 瓶頸：？ 踩坑：？ 浪費率：？
```

### Phase 2：預判（每條路徑的 Pre-mortem）

對每條路徑問五個問題：

1. **瓶頸在哪？** — 依賴什麼外部資源？哪步會卡最久？
2. **為什麼會失敗？** — 這條路徑最可能的失敗模式是什麼？
3. **歷史教訓？** — 我們之前做過類似的事嗎？踩過什麼坑？
4. **勘探/建設比率？** — 這條路徑有多少時間會花在非建設性工作上？
5. **如果必須選這條路徑，怎麼降低風險？** — 最小化傷害的策略

### Phase 3：收斂（風險地圖 + 推薦）

輸出格式：

```
🔮 Pre-mortem 路徑分析 — {任務摘要}

📊 路徑風險地圖：
  路徑A（直接法）：🏗️30% 🔍70% ⚠️高風險 — 已知瓶頸：[X]
  路徑B（替代法）：🏗️60% 🔍40% ⚠️中風險 — 避開了 [X]
  路徑C（極簡法）：🏗️80% 🔍20% ⚠️低風險 — 但可能遺漏 [Y]
  路徑D（反向法）：🏗️40% 🔍60% ⚠️中風險
  路徑E（詢問法）：🏗️100% 🔍0% ⚠️最低 — 但依賴用戶輸入

🎯 推薦：路徑 C + B 混合
  原因：[說明]

⚠️ 關鍵預判：
  • [預判1] — 如果發生，切換到路徑X
  • [預判2] — 如果發生，暫停並詢問

💬 選擇路徑或自定義混合
```

### Phase 4：執行（選定路徑後）

- 按選定路徑執行
- 中途如果預判的風險信號出現 → 自動觸發重新評估
- 完成後對比預判 vs 實際 → 寫入 lessons（強化預判能力）
